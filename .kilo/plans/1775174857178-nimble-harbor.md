# PLAN: Fix User Registration - Missing Database Columns

## Problem Summary
User registration fails with "Failed to fetch" error because:
1. **Database schema incomplete**: `users` table missing critical authentication columns
2. **Existing database prevents auto-migration**: SQLAlchemy `create_all()` won't modify existing tables

## Root Cause Analysis

### Current Database State
**Location**: `/Users/ramzi/Documents/filpIQ antigravity openrouter/flipiq.db`
**Created**: April 2, 2026 22:43

**Current `users` table schema (INCOMPLETE)**:
```sql
CREATE TABLE users (
    id INTEGER NOT NULL,
    email VARCHAR NOT NULL,
    "plan" VARCHAR,
    stripe_customer_id VARCHAR,
    stripe_subscription_id VARCHAR,
    created_at DATETIME,
    updated_at DATETIME,
    PRIMARY KEY (id)
);
```

**Missing columns**:
- `hashed_password` (required for auth)
- `name` (user display name)
- `is_verified` (email verification status)
- `is_active` (account active status)
- `last_login` (track user sessions)

### Expected Schema (from models.py)
```python

    updated_at = Column(DateTime, default=lambda: datetime.now(timezone.utc), onupdate=lambda: datetime.now(timezone.utc))
```

### Why This Happened

1. **Database created before Phase 3 auth implementation**: The database was created in earlier phases when `users` table only needed basic fields (email, plan, stripe IDs)

2. **SQLAlchemy limitation**: `Base.metadata.create_all(bind=engine)` only creates tables that don't exist. It does NOT:
   - Add new columns to existing tables
   - Modify existing column definitions
   - Run migrations

3. **Absolute path change**: The database path was recently changed to absolute (line 7 of database.py) to prevent "database switching" issues, but this means we're now using the same database file consistently

## Solution Options

### Option A: Add Missing Columns with ALTER TABLE (RECOMMENDED)
**Pros**: Preserves all existing data, quick fix
**Cons**: Requires manual SQL execution

**Steps**:
1. Stop backend server
2. Backup database
3. Add missing columns using ALTER TABLE
4. Restart backend
5. Test registration

**SQL Commands**:
```sql
ALTER TABLE users ADD COLUMN hashed_password VARCHAR;
ALTER TABLE users ADD COLUMN name VARCHAR;
ALTER TABLE users ADD COLUMN is_verified INTEGER DEFAULT 0;
ALTER TABLE users ADD COLUMN is_active INTEGER DEFAULT 1;
ALTER TABLE users ADD COLUMN last_login DATETIME;
```

**Estimated time**: 2-3 minutes

### Option B: Drop and Recreate Users Table
**Pros**: Clean schema, guaranteed to match model
**Cons**: Loses all existing user data (if any)

**Steps**:
1. Stop backend server
2. Backup database
3. Drop users table: `DROP TABLE users;`
4. Restart backend (init_db will recreate with correct schema)
5. Test registration

**Estimated time**: 1-2 minutes

### Option C: Create New Database
**Pros**: Fresh start, no schema conflicts
**Cons**: Loses ALL data (auctions, items, settings, etc.)

**Steps**:
1. Stop backend server
2. Rename existing database: `mv flipiq.db flipiq.db.backup`
3. Restart backend (creates fresh database)
4. Test registration

**Estimated time**: 1-2 minutes + data loss

## Recommended Solution: Option A (ALTER TABLE)

**Reason**: 
- Preserves all existing data (auctions, items, auction houses, settings)
- No risk of data loss
- Quick and safe
- Matches the production database migration approach
