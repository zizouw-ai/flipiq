# PLAN: Fix User Data Isolation

## Problem Statement

**Issue**: New user (user1@test.com) can access auctions created by dev user. No data separation between users.

**Impact**: Security vulnerability - users can see/modify other users' data.

## Root Cause Analysis

### Database Investigation Findings

**Current state**:
- 1 user in database: `id=1, email=user1@test.com`
- 6 auctions in database: auctions 1-4 have `user_id=1`, auctions 5-6 have `user_id=NULL`
- 1 item in database: `id=1, auction_id=5, user_id=1`

**The problem**: 
1. Auctions 5-6 have `user_id=NULL` (orphaned records)
2. User isolation code IS correctly implemented in `auctions.py`:
   - Line 60: `filter(Auction.user_id == current_user.id)` on list_auctions
   - Line 88: `filter(Auction.id == auction_id, Auction.user_id == current_user.id)` on get_auction
   - All other endpoints properly filter by user

3. **The dev-mode bypass**: In `jwt.py:172-181`, when `token="dev-token"`, the system returns a mock user with `id=1`. This means:
   - Dev user gets user_id=1
   - Real user also got user_id=1 (first registered user)
   - They appear to be the SAME USER to the database!

### Why This Happened

1. **Registration created user with id=1**: First real user got id=1
2. **Dev mode also uses id=1**: The mock dev user hardcoded to id=1
3. **Existing data from dev mode**: Auctions created during dev mode have `user_id=NULL` or `user_id=1`
4. **No proper user assignment during creation**: Some auctions/items created without user_id

### Authentication Flow Analysis

**Frontend** (`authStore.js`):
- Line 78-84: Dev mode enabled with `enableDevMode()` sets `token='dev-token'`
- Line 89-94: `getAuthHeaders()` sends `Authorization: Bearer dev-token` or real token
- Dev mode bypass intended for testing, but created data ownership issues

**Backend** (`jwt.py`):
- Line 172: Dev token check: `if token == "dev-token"`
- Line 174-181: Returns mock user with `id=1`
- This id=1 clashes with the first real registered user!

## Solution Strategy

### Option A: Fix Data + Prevent Dev Mode ID Collision (RECOMMENDED)

**Pros**: 
- Preserves all data
- Proper user isolation
- Fixes root cause
- Prevents future issues

**Cons**: Requires multiple steps

### Option B: Wipe Database and Start Fresh

**Pros**: Quick, clean slate

**Cons**: Loses all test data, doesn't fix the underlying dev mode issue

## Recommended Solution: Option A

### Phase 1: Create Proper Dev User

1. **Create dedicated dev user in database**:
   ```sql
   INSERT INTO users (email, hashed_password, name, plan, is_verified, is_active) 
   VALUES ('dev@flipiq.local', '<dev-password-hash>', 'Developer', 'pro', 1, 1);
   ```
   This will get `id=2` (or next available)

2. **Update dev mode to use dedicated user**:
   - Change `jwt.py:175` from `id=1` to query database for `dev@flipiq.local`
   - Or assign a high ID like `id=999999` that won't conflict

### Phase 2: Assign Orphaned Data to Dev User

1. **Identify orphaned records**:
   ```sql
   SELECT * FROM auctions WHERE user_id IS NULL;
   SELECT * FROM items WHERE user_id IS NULL;
   ```

2. **Assign to dev user** (after creating dev user with id=2):
   ```sql
   UPDATE auctions SET user_id = 2 WHERE user_id IS NULL;
   UPDATE items SET user_id = 2 WHERE user_id IS NULL;
   ```

### Phase 3: Verify User Isolation

1. **Test with curl**:
   ```bash
   # Login as user1@test.com
   curl -X POST http://localhost:8000/api/auth/login \
     -H "Content-Type: application/json" \
     -d '{"email":"user1@test.com","password":"Test123!"}'
   
   # List auctions (should only see user_id=1 auctions)
   curl -X GET http://localhost:8000/api/auctions/ \
     -H "Authorization: Bearer <user1_token>"
   ```

2. **Test with dev mode**:
   - Enable dev mode
   - List auctions (should see dev user auctions only)

### Phase 4: Update Code to Prevent Future Issues

**File: `backend/app/auth/jwt.py`**

Change dev mode implementation (lines 172-181):

```python
def require_auth(
    token: str = Depends(oauth2_scheme),
    db: Session = Depends(get_db)
) -> User:
    # Check for dev mode token
    if token == "dev-token":
        # Look for dev user in database, create if not exists
        dev_user = db.query(User).filter(User.email == "dev@flipiq.local").first()
        if not dev_user:
            # Create dev user with high ID to avoid conflicts
            dev_user = User(
                id=999999,  # High ID won't conflict with real users
                email="dev@flipiq.local",
                name="Developer",
                plan="pro",
                is_active=1,
                is_verified=1,
            )
            db.add(dev_user)
            db.commit()
        return dev_user
    
    # Normal auth flow...
```

**Alternative**: Remove dev mode entirely and require real authentication.

### Phase 5: Fix Middleware Limits (Bonus Issue Found)

**File: `backend/app/middleware/limits.py`**

**Issue**: `get_item_count()` returns global count, not per-user count.

**Fix**: Update to accept user_id parameter:

```python
def get_item_count(db: Session, user_id: Optional[int] = None) -> int:
    """Get the total count of items for a user."""
    if user_id:
        return db.query(Item).filter(Item.user_id == user_id).count()
    return db.query(Item).count()
```

Update callers in `auctions.py:132`:
```python
current_item_count = get_item_count(db, current_user.id)
```
