# PLAN: Fix "Add New Auction" Failure

## Problem Statement
Users (both real and dev mode) cannot create new auctions. When clicking "Save Auction" in the modal:
1. The modal closes immediately.
2. No auction is saved to the database.
3. The user is not properly notified of the specific error.

## Root Cause Analysis

### 1. Frontend "Silent" Failure
In `frontend/src/pages/AuctionTracker.jsx`, the `createAuction` handler is structured as follows:
```javascript
const createAuction = async () => {
  try {
    await api.createAuction({ ...aForm, total_hammer: parseFloat(aForm.total_hammer) || 0 });
    setShowForm(false); // <<------ Modal closes here
    setAForm({ ... });
    loadAuctions();
  } catch (err) {
    alert('Failed to create auction: ' + err.message);
  }
};
```
While the `setShowForm(false)` is inside the `try` block, if the API returns a non-OK response, the `api.createAuction` call throws an error. However, if the error is not handled perfectly or if there's a race condition, the user perceives it as "nothing happened" because the UI state might be inconsistent or the alert is missed.

### 2. API Header Missing `Content-Type`
In `frontend/src/api.js`, the `request` helper (lines 7-24) merges headers from `useAuthStore.getState().getAuthHeaders()`. 
- The `getAuthHeaders` function in `authStore.js` provides `Content-Type: application/json` **only if a token exists**.
- If for any reason the token is missing or the store is in a weird state, the request is sent without `Content-Type: application/json`.
- FastAPI returns a **422 Unprocessable Entity** if the `Content-Type` is missing for a JSON body.

### 3. Backend Validation (Pydantic)
The `AuctionCreate` schema in `backend/app/schemas.py` requires:
- `name: str`
- `date: str`
If the frontend sends these as empty strings (which they are initialized as in `aForm`), the backend might reject them if the Pydantic model expects non-empty strings (though `str` usually allows `""`).

### 4. Authentication Collision (Linked to Isolation Issue)
As identified in the isolation plan, the `dev-token` maps to `user_id=1`, and the first registered user also has `user_id=1`. While this doesn't block the *creation* of a record, it causes the "User A sees User B's data" problem.

---

## Solution Strategy

### Phase 1: Frontend Robustness
1. **Explicit Headers**: Modify `frontend/src/api.js` to ensure `Content-Type: application/json` is ALWAYS sent for `POST`/`PUT` requests, regardless of auth state.
2. **UI Feedback**: Update `frontend/src/pages/AuctionTracker.jsx` to ensure the modal only closes on **success**, and provides a clear error message on failure.

### Phase 2: Backend Validation & Logging
1. **Enhanced Logging**: Add detailed logging to `backend/app/routers/auctions.py` in the `create_auction` endpoint to capture exactly what payload is received and why it might be failing (e.g., logging the `current_user` object).

---

## Implementation Steps

### Step 1: Fix `frontend/src/api.js`
Update the `request` function to guarantee JSON headers for writes.
```javascript
// Current
headers: { ...authHeaders, ...options.headers },

// Fixed
headers: { 
  'Content-Type': 'application/json', 
  ...authHeaders, 
  ...options.headers 
},
```

### Step 2: Fix `frontend/src/pages/AuctionTracker.jsx`
Move `setShowForm(false)` to occur only after successful API response.
```javascript
const createAuction = async () => {
  try {
    await api.createAuction({ ...aForm, total_hammer: parseFloat(aForm.total_hammer) || 0 });
    setShowForm(false); // Success path
    // ... rest of success logic
  } catch (err) {
    alert('Failed to create auction: ' + err.message);
    // Do NOT close the form here so user can fix the input
  }
};
```

### Step 3: Verify and Test
1. **Test New User**: Register a new user $\rightarrow$ Create auction $\rightarrow$ Verify it appears in list.
2. **Test Dev Mode**: Enable dev mode $\rightarrow$ Create auction $\rightarrow$ Verify it appears.
3. **Network Inspection**: Verify the `POST /api/auctions/` request has `Content-Type: application/json` and returns `200 OK`.
