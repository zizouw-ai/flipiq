# Implementation Plan: FlipIQ Deployment & Compatibility Fixes

This plan addresses the critical deployment errors (405s), database compatibility issues, and authentication gaps identified in the laest audit.

## 🎯 Goals
1. Fix the production routing issue (405 Method Not Allowed).
2. Ensure all write-operations are user-scoped and authenticated.
3. Remove hardcoded local paths to ensure PostgreSQL/Railway compatibility.
4. Align database models with PostgreSQL standards.

---

## 🛠️ Execution Steps

### Step 1: Production Routing Fix (P0)
**Target:** `railway.toml`
- Change `${{RAILWAY_SERVICE_FLIPIQ_BACKEND_URL}}` to `${RAILWAY_SERVICE_FLIPIQ_BACKEND_URL}` in line 26.
- **Verification:** Deploy to Railway and verify POST requests to `/api/auctions/` return 200/201.

### Step 2: Database & Environment Portability (P1)
**Target:** `backend/app/database.py`
- Replace hardcoded Mac path `/Users/ramzi/Documents/...` in `DEFAULT_DB_PATH` with `os.getenv("RAILWAY_VOLUME_MOUNT_PATH", ".")`.
- **Verification:** Confirm app starts without crashing when `DATABASE_URL` is absent.

**Target:** `backend/app/models.py`
- Remove `autoincrement=True` from all primary key columns (User, AuctionHouseConfig, ShippingPreset, ItemTemplate).
- **Verification:** Run tests to ensure primary keys still generate correctly.

### Step 3: Authentication & User Scoping (P1)
**Target:** `backend/app/routers/shipping_presets.py`
- Add `current_user: User = Depends(require_auth)` to `POST`, `PUT`, and `DELETE` endpoints.
- In `POST`, ensure `user_id=current_user.id` is passed to the model creation.

**Target:** `backend/app/routers/settings.py`
- Add `current_user: User = Depends(require_auth)` to all endpoints.
- Update logic to filter and save settings using `user_id`.

**Target:** `backend/app/routers/limits.py`
- Replace the mock `get_current_user` function with the actual `require_auth` dependency.

### Step 4: Quality Assurance & Testing (P2)
- **Create `backend/tests/test_auth.py`**: Implement tests for registration, login, and protected route access.
- **Verify PostgreSQL Compatibility**: Run the full test suite using a PostgreSQL instance if available, or verify schema via SQLAlchemy.
- **Full Suite Run**: Run `pytest backend/tests/ -v` and ensure 173+ tests pass.

## ⚠️ Risks & Mitigations
- **Risk:** Modifying `models.py` might cause data loss in existing SQLite DBs if IDs are not handled correctly.
- **Mitigation:** Perform a full DB backup before applying model changes.
- **Risk:** Updating `require_auth` on settings might break existing frontend calls.
- **Mitigation:** Update `frontend/src/api.js` if any endpoints were previously assumed to be public.
