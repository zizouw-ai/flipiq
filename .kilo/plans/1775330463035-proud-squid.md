# Plan: Debug Railway Deployment Issues

## Problem Statement
After deploying to Railway, the application is completely non-functional:
- Cannot login with previously registered users
- Form submissions don't work (buttons do nothing)
- Calculations fail
- Only dev mode works, but even then nothing functions
- Tests pass locally but production is broken

This indicates a fundamental disconnect between local dev environment and Railway deployment.

## Root Cause Analysis Needed

### Hypothesis 1: Database Not Persisting on Railway
**Symptoms**: Users can't login, data isn't saved
**Likelihood**: HIGH
**Evidence**:
- Railway Volume mounted at `/data`
- SQLite database file may not be at correct path
- Database might be reset on each deployment

### Hypothesis 2: Environment Variables Missing
**Symptoms**: Auth fails, buttons don't work
**Likelihood**: HIGH
**Evidence**:
- `JWT_SECRET_KEY` must be set consistently
- Frontend `VITE_API_URL` might be incorrect
- CORS origins may be blocking requests

### Hypothesis 3: Frontend-Backend Communication Broken
**Symptoms**: Buttons don't work, no feedback
**Likelihood**: MEDIUM
**Evidence**:
- Railway service URLs might not be resolving correctly
- CORS might be blocking requests
- API base URL mismatch

### Hypothesis 4: Auth Token Issues
**Symptoms**: Can't login, dev mode only works
**Likelihood**: MEDIUM
**Evidence**:
- Token validation requires consistent SECRET_KEY
- Token might be invalid due to clock skew
- User data might not exist in production DB

## Investigation Steps

### Step 1: Check Railway Environment Variables
**Action**: Verify all required env vars are set in Railway dashboard
**Required Variables**:
- `JWT_SECRET_KEY` (MUST be consistent across deploys)
- `DATABASE_URL` (or confirm SQLite path)
- `RAILWAY_VOLUME_MOUNT_PATH` (should be `/data`)
- `CORS_ORIGINS` (frontend URL)
- `VITE_API_URL` (backend URL - set at build time)

### Step 2: Verify Database Path
**Action**: Check where SQLite file is actually created
**Current Config**: `database.py` line 7-8
```python
DEFAULT_DB_PATH = os.getenv("RAILWAY_VOLUME_MOUNT_PATH", ".")
DATABASE_URL = os.getenv("DATABASE_URL", f"sqlite:///{DEFAULT_DB_PATH}/flipiq.db")
```
**Problem**: If Railway doesn't provide `DATABASE_URL` or `RAILWAY_VOLUME_MOUNT_PATH`, DB is at `./flipiq.db` which may not persist.

### Step 3: Check Frontend API URL
**Action**: Verify frontend is calling correct backend URL
**Current Config**: `railway.toml` line 26, 32
```toml
BACKEND_URL = "https://${RAILWAY_SERVICE_FLIPIQ_BACKEND_URL}/api"
VITE_API_URL = "https://${RAILWAY_SERVICE_FLIPIQ_BACKEND_URL}/api"
```
**Problem**: `${RAILWAY_SERVICE_FLIPIQ_BACKEND_URL}` is Railway's reference syntax. Need to verify Railway actually substitutes this.

### Step 4: Check CORS Configuration
**Action**: Verify CORS allows frontend domain
**Current Config**: `main.py` lines 34-67
**Potential Issue**: If frontend URL isn't in `CORS_ORIGINS`, all requests will be blocked.

## Recommended Fixes

### Fix 1: Explicit Database Path for Railway
**File**: `backend/app/database.py`
**Change**: Force SQLite to use `/data/flipiq.db` on Railway
```python
import os

# Railway Volume is at /data
RAILWAY_VOLUME = os.getenv("RAILWAY_VOLUME_MOUNT_PATH", "")
DATABASE_URL = os.getenv("DATABASE_URL")

if DATABASE_URL:
    # PostgreSQL in production (Phase 3)
    pass
elif RAILWAY_VOLUME:
    # Railway with volume mount
    DATABASE_URL = f"sqlite:///{RAILWAY_VOLUME}/flipiq.db"
else:
    # Local development
    DATABASE_URL = "sqlite:///./flipiq.db"
```

### Fix 2: Set Consistent JWT_SECRET_KEY
**File**: Railway Dashboard → flipiq-backend → Variables
**Action**: Add variable `JWT_SECRET_KEY` with a strong random value
**Important**: This MUST be set and must remain the same across all deploys

### Fix 3: Add Debug Endpoint
**File**: `backend/app/main.py`
**Add**: Endpoint to check database and environment status
```python
@app.get("/debug/status")
def debug_status():
    import os
    from app.database import engine
    return {
        "database_url": DATABASE_URL.split("@")[-1] if "@" in DATABASE_URL else DATABASE_URL,
        "volume_mount": os.getenv("RAILWAY_VOLUME_MOUNT_PATH", "NOT_SET"),
        "db_file_exists": os.path.exists(f"{RAILWAY_VOLUME}/flipiq.db") if RAILWAY_VOLUME else os.path.exists("./flipiq.db"),
        "cors_origins": CORS_ORIGINS,
        "jwt_secret_set": bool(os.getenv("JWT_SECRET_KEY")),
    }
```

## How to Collect Debug Info

### Step A: Get Railway Logs
Run this command in your terminal:
```bash
railway logs --service flipiq-backend --tail 50
```
If that doesn't work, try:
```bash
railway logs
```
Then copy-paste the last 50 lines here.

### Step B: Check Railway Environment Variables
1. Go to https://railway.app/dashboard
2. Click on your `flipiq-backend` service
3. Click "Variables" tab in the left sidebar
4. Screenshot or list all variables set

### Step C: Test Backend Health
First, find your backend URL from Railway dashboard (it looks like `https://flipiq-backend-production-abc123.up.railway.app`)

Then run:
```bash
curl https://YOUR-BACKEND-URL.up.railway.app/health
```
Replace `YOUR-BACKEND-URL` with the actual URL from Railway.

### Step D: Check Browser Console
1. Open your Railway frontend URL in Chrome/Firefox
2. Press F12 to open Developer Tools
3. Click "Console" tab
4. Try to login or click a button
5. Copy any red error messages

Share the results from ALL 4 steps above.
